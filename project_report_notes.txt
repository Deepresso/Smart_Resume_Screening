--------------------------------------------------------------------------------
1. SYSTEM OVERVIEW
--------------------------------------------------------------------------------
The Smart Resume Screening System is a full-stack web application that automates
the resume screening process using Natural Language Processing (NLP) techniques.
The system serves two types of users:

  - HR Manager : Posts job vacancies, reviews screened candidates, shortlists
                 or rejects applicants based on NLP-generated scores.

  - Applicant  : Registers an account, browses active job postings, uploads
                 a resume (PDF or DOCX), and views their match scores.

When an applicant submits a resume, the system automatically runs four NLP
algorithms and produces a composite relevance score. HR can then view all
applicants ranked from highest to lowest score.

4. DATABASE DESIGN
--------------------------------------------------------------------------------
Three tables are used, defined in models.py using SQLAlchemy ORM.

TABLE: users
  id         INTEGER PRIMARY KEY
  name       VARCHAR(100)   Full name of the user
  email      VARCHAR(150)   Unique login email
  password   VARCHAR(255)   bcrypt-hashed password
  role       VARCHAR(20)    Either 'hr' or 'applicant'

TABLE: job_postings
  id          INTEGER PRIMARY KEY
  title       VARCHAR(150)   Job title
  company     VARCHAR(150)   Company name
  location    VARCHAR(100)   Job location
  description TEXT           Full job description (used for TF-IDF and BERT)
  keywords    TEXT           Comma-separated screening keywords
  created_by  INTEGER        Foreign key to users.id
  created_at  DATETIME       Timestamp of creation
  is_active   BOOLEAN        Whether the posting is open or closed

TABLE: applications
  id               INTEGER PRIMARY KEY
  user_id          INTEGER    Foreign key to users.id
  job_id           INTEGER    Foreign key to job_postings.id
  resume_path      VARCHAR    Filename of the uploaded resume
  resume_text      TEXT       Extracted plain text from the resume
  cover_letter     TEXT       Optional cover letter text
  keyword_score    FLOAT      Score from Keyword Matching algorithm
  fuzzy_score      FLOAT      Score from Fuzzy Matching algorithm
  similarity_score FLOAT      Score from TF-IDF Cosine Similarity
  semantic_score   FLOAT      Score from BERT Semantic Similarity
  composite_score  FLOAT      Weighted final score
  status           VARCHAR    'submitted', 'shortlisted', or 'rejected'
  applied_at       DATETIME   Timestamp of submission

6. JOB POSTING MANAGEMENT
--------------------------------------------------------------------------------
Implementation file: app.py (routes), templates/hr/

HR can perform full CRUD on job postings:

  CREATE : Fill in job title, company, location, description, and add
           screening keywords using a tag-style input (JavaScript-driven).
           Keywords are stored as a comma-separated string in the database.

  READ   : All job postings are listed in a table showing title, applicant
           count, status (active/closed), and date posted.

  UPDATE : HR can edit any field of an existing job posting. The existing
           keywords are pre-loaded as tags in the edit form.

  DELETE : Deleting a job also deletes all related applications first
           (to avoid foreign key constraint errors in SQLite).

  TOGGLE : HR can close or reopen a job posting. Closed jobs do not
           appear in the applicant's Browse Jobs page.

Keyword Input (JavaScript):
  - Keywords are entered one at a time and displayed as removable tag chips.
  - Before form submission, all keyword tags are collected into a hidden
    input field as a comma-separated string.
  - The Enter key triggers keyword addition without submitting the form.

7. RESUME UPLOAD AND TEXT EXTRACTION
--------------------------------------------------------------------------------
Implementation file: screening.py (extract_text), app.py (applicant_apply route)

Upload Process:
  1. Applicant selects a PDF or DOCX file (max 10MB).
  2. The file is validated for allowed extensions (pdf, docx).
  3. The file is saved to the /uploads/ folder with a unique filename
     format: {user_id}_{job_id}_{original_filename}
  4. Text is extracted from the file immediately after saving.
  5. NLP scoring runs on the extracted text.
  6. All scores and the extracted text are saved to the applications table.

Text Extraction:
  - PDF files  : PyMuPDF (fitz) iterates through each page and concatenates
                 the text content.
  - DOCX files : python-docx reads each paragraph and joins non-empty lines.

Duplicate prevention:
  - A check is performed before saving. If the applicant has already applied
    to the same job, the submission is rejected with a flash message.

8. NLP SCREENING ENGINE
--------------------------------------------------------------------------------
Implementation file: screening.py

The system uses FOUR NLP techniques to score each resume against a job posting.
All functions are in screening.py and called from compute_scores().

------------------------------------
8.1 KEYWORD MATCHING (Weight: 30%)
------------------------------------
Function: keyword_score(resume_text, keywords)

How it works:
  - The resume text and each keyword are both converted to lowercase.
  - For each keyword in the job's keyword list, the system checks if the
    keyword appears anywhere in the resume text using Python's 'in' operator.
  - Score = (number of keywords found / total keywords) x 100

Example:
  Job keywords: Python, Flask, SQL, Docker
  Resume contains: Python, Flask, SQL
  Score = 3/4 x 100 = 75%

Strength  : Simple, transparent, exact matching
Limitation: Does not catch synonyms or spelling variations

------------------------------------
8.2 FUZZY MATCHING (Weight: 25%)
------------------------------------
Function: fuzzy_score(resume_text, keywords)
Library  : rapidfuzz

How it works:
  - The resume text is tokenized into individual words.
  - For each keyword, the system computes the Levenshtein ratio between
    the keyword and every word in the resume, then takes the best match.
  - The average of all keyword best-match scores is the fuzzy score.
  - Uses fuzz.ratio() which measures character-level edit distance.

Example:
  Keyword: "Python"
  Resume word: "Pyhton" (typo)
  fuzz.ratio("python", "pyhton") = ~91% — still detected as a strong match

Strength  : Handles typos, abbreviations, and minor spelling variations
Limitation: Can produce false positives with short words

------------------------------------
8.3 TEXT SIMILARITY — TF-IDF (Weight: 25%)
------------------------------------
Function: similarity_score(resume_text, job_description)
Library  : scikit-learn

How it works:
  - TF-IDF (Term Frequency-Inverse Document Frequency) converts each
    document (resume and job description) into a numerical vector.
  - TF measures how often a word appears in a document.
  - IDF reduces the weight of common words that appear in both documents.
  - Cosine similarity measures the angle between the two TF-IDF vectors.
  - A score of 100% means the documents are identical in word distribution.
  - No training is required — the vectorizer is built fresh for each pair.

Strength  : Captures overall thematic alignment between documents
Limitation: Treats words as independent tokens, ignores word meaning and context

------------------------------------
8.4 COMPOSITE SCORE FORMULA (3 techniques)
------------------------------------
composite = (keyword x 0.40) + (fuzzy x 0.30) + (tfidf x 0.30)

Rationale for weights:
  - Keyword Matching gets the highest weight (40%) because exact keyword
    presence is the most direct signal of candidate qualification.
  - Fuzzy and TF-IDF each get 30% as complementary surface-level techniques.

Note — BERT semantic scoring was implemented and tested during development
but removed from the final system due to deployment constraints:
  - The sentence-transformers library (PyTorch-based) requires ~350MB RAM.
  - Render's free tier only provides 512MB RAM total.
  - Loading the BERT model during a request caused Linux OOM-killer to
    send SIGKILL to the worker process (~36 seconds in), resulting in a
    502 Bad Gateway error for the applicant.
  - Multiple workarounds were attempted (HuggingFace Inference API via HTTP)
    but Render's free tier showed intermittent DNS resolution failures for
    external ML hosting services.
  - Final decision: remove BERT and use the reliable 3-algorithm system.
  - semantic_score column is retained in the database (always stores 0.0)
    to avoid breaking existing data.

All scores are stored in the database and displayed to HR on the
Screening Results page, ranked from highest to lowest composite score.

14. KEY DESIGN DECISIONS
--------------------------------------------------------------------------------

Why SQLite?
  SQLite was chosen for development because it requires no separate database
  server, stores data in a single file, and is fully supported by SQLAlchemy.
  Switching to PostgreSQL or MySQL only requires changing one line in app.py:
    app.config['SQLALCHEMY_DATABASE_URI'] = 'postgresql://...'

Why store extracted resume text in the database?
  Storing resume_text in the applications table means:
    - NLP can be re-run without re-reading the file.
    - HR can read the resume content directly in the browser.
    - The file itself only needs to be kept for download purposes.

Why not use Flask-Migrate for schema changes?
  Flask-Migrate (Alembic) adds complexity for a single-developer FYP project.
  Instead, a lightweight auto-migration block checks for missing columns on
  startup and applies ALTER TABLE if needed. This is sufficient for SQLite
  and the scale of this project.

Why applicant-only registration?
  Allowing self-registration for HR would create a security vulnerability
  where anyone could create an HR account and access all candidate data.
  The HR account is seeded automatically by the system on first startup.

15. EMAIL VERIFICATION SYSTEM
--------------------------------------------------------------------------------
Library     : SendGrid Python SDK 6.12.5 (HTTP API)
Sender      : resumatch.fyp@gmail.com (verified on SendGrid Single Sender)

Why SendGrid instead of Flask-Mail / Gmail SMTP:
  Render's free tier blocks all outbound SMTP connections (ports 587 and 465)
  to prevent spam abuse. Flask-Mail with Gmail SMTP caused worker timeouts.
  SendGrid uses an HTTP API instead of SMTP, which is not blocked by Render.
  Free tier: 100 emails/day — sufficient for FYP purposes.

Async email sending:
  Email is sent in a background Python thread (threading.Thread, daemon=True)
  so the registration request completes immediately without waiting for the
  SendGrid API call. This prevents request timeouts on Render's free tier.

Registration flow with email verification:
  1. Applicant fills in the registration form (name, email, phone, address, password)
  2. System validates Malaysian phone format using regex: ^(\+?60|0)1[0-9]\d{7,8}$
     - Accepted: 0123456789, 01234567890, +60123456789
     - Rejected: anything that does not match Malaysian mobile number format
  3. Duplicate email and duplicate phone checks — blocked with flash message
  4. A random 4-digit code is generated (random.randint(1000, 9999))
  5. The code is stored in the users.verify_code column
  6. Account is saved with is_verified=False (blocked from login)
  7. Verification email sent asynchronously via SendGrid HTTP API
  8. Applicant is redirected to /register/success (popup-style page)
  9. User clicks "OK, Go to Sign In" → redirected to /login

Verification page (/verify):
  - Accessed when an unverified user tries to log in
  - Shows the email address the code was sent to
  - Single 4-digit numeric input (auto-filters non-digits via JavaScript)
  - If code matches users.verify_code: sets is_verified=True, clears code, logs in
  - If code is wrong: shows error, lets them try again
  - Resend Code button: generates a new code, saves to DB, sends new email async

Login guard:
  - On login, if is_verified=False: redirects to /verify instead of logging in
  - Session stores pending_verify_id to identify which user is verifying

Existing accounts (HR admin, pre-migration users):
  - Auto-migration adds is_verified column with DEFAULT TRUE (PostgreSQL) or
    DEFAULT 1 (SQLite), so all existing accounts are automatically verified
    and not blocked from logging in

New profile fields added to users table:
  - phone       VARCHAR(20)   Malaysian mobile number
  - address1    VARCHAR(200)  Address line 1 (required)
  - address2    VARCHAR(200)  Address line 2 (optional)
  - postcode    VARCHAR(10)   5-digit Malaysian postcode
  - state       VARCHAR(50)   Malaysian state (dropdown)
  - is_verified BOOLEAN       Whether email has been verified
  - verify_code VARCHAR(6)    Temporary 4-digit verification code

Registration success flow:
  - After successful registration, user is redirected to /register/success
  - A clean success page shows a popup-style card: "Verification Email Sent!"
  - User clicks "OK, Go to Sign In" to go to the login page
  - On login with unverified account: redirected to /verify page to enter code
  - This separates registration from verification (cleaner UX)

Duplicate checks on registration:
  - Duplicate email: blocked with flash message
  - Duplicate phone number: blocked with flash message

HR User Management (admin panel):
  - Route: /hr/users — list all users (HR and applicants)
  - Route: /hr/users/<id>/edit — edit any user's details and verification status
  - Route: /hr/users/<id>/delete — delete an applicant account (and all their applications)
  - HR accounts are protected from deletion
  - HR can manually set is_verified=True to activate accounts (useful when email fails)
  - Located in HR portal sidebar under "Management"

Security note:
  - Gmail App Password used instead of account password (Google security policy)
  - Credentials stored in environment variables, never hardcoded in source code
  - .env file used for local development (excluded from git via .gitignore)
  - Render environment variables used for production

16. DEPLOYMENT
--------------------------------------------------------------------------------
Platform    : Render (render.com)
Repository  : https://github.com/Deepresso/Smart_Resume_Screening
Database    : PostgreSQL (Render free tier, replaces SQLite in production)

Files added for deployment:
  requirements.txt  - All Python dependencies with pinned versions
  Procfile          - Tells Render how to start the app: gunicorn app:app
  render.yaml       - Full deployment config (web service + PostgreSQL database)

Key deployment changes in app.py:
  - SECRET_KEY is read from environment variable (Render auto-generates a secure one)
  - DATABASE_URL is read from environment variable (Render injects this automatically)
  - Render provides DATABASE_URL as postgres:// — app.py converts it to postgresql://
    because SQLAlchemy 2.x requires the postgresql:// scheme

How auto-deployment works:
  1. Code is pushed to GitHub (main branch)
  2. Render detects the push and triggers a rebuild
  3. Render runs: pip install -r requirements.txt
  4. Render starts the app with: gunicorn app:app
  5. The app is accessible at the Render-assigned URL

Production vs Local difference:
  - Local  : Uses SQLite (instance/smart_resume.db), started with py app.py
  - Render : Uses PostgreSQL, started with gunicorn, env vars injected by Render

Environment variables required on Render:
  - SECRET_KEY       : Auto-generated by Render (via render.yaml)
  - DATABASE_URL     : Auto-injected by Render PostgreSQL (via render.yaml)
  - SENDGRID_API_KEY : SendGrid HTTP API key for sending verification emails
  - MAIL_USERNAME    : resumatch.fyp@gmail.com (sender email address)

PostgreSQL connection stability:
  - Added SQLALCHEMY_ENGINE_OPTIONS = {'pool_pre_ping': True}
  - pool_pre_ping tests the connection before each query and reconnects
    if the connection was dropped (common on Render free tier after idle)
  - Without this, the app returns 500/502 errors on the first request
    after the database connection goes idle

Known limitations:
  - Render free tier spins down after 15 minutes of inactivity.
    The first request after idle takes ~30 seconds to wake up.
  - Uploaded resume files are stored on Render's ephemeral filesystem.
    They persist during a session but are cleared on redeploy. For production,
    files should be stored in cloud storage (e.g. AWS S3 or Cloudinary).
    For FYP demo purposes this is acceptable.
  - Single gunicorn worker (--workers 1) means only one resume submission is
    processed at a time. Concurrent submissions are queued. Acceptable for FYP.

gunicorn:
  - Gunicorn is a production-grade WSGI server for Python.
  - Flask's built-in development server (py app.py) is not suitable for
    production as it handles only one request at a time.
  - Gunicorn handles concurrent requests and is the standard for deploying
    Flask applications on cloud platforms.

psycopg2-binary:
  - The PostgreSQL adapter for Python. Required by SQLAlchemy to connect to
    a PostgreSQL database. The binary version includes all native dependencies
    pre-compiled so no system libraries need to be installed separately.

  - Form rows (form-row) stack vertically instead of side-by-side
  - Application progress timeline wrapped in .timeline-scroll div for
    horizontal scrolling (min-width: 480px on inner container)
  - Auth pages: left panel hidden, form takes full width

Very small (≤380px):
  - Stats collapse to 1 column
  - Further padding reduction

17. LIMITATIONS AND FUTURE IMPROVEMENTS
--------------------------------------------------------------------------------

Current Limitations:
  1. Only one HR account is supported (the seeded admin account).
  2. Existing applications do not get re-scored when job keywords are edited.
  3. The system does not send email notifications for shortlisted applicants.
  4. Uploaded resume files are lost on Render redeploy (ephemeral filesystem).
  5. Only one concurrent resume submission at a time (single gunicorn worker).
  6. BERT semantic scoring was removed due to Render free tier RAM constraints
     (see Section 8.4). Running on a paid server would re-enable it.

Possible Future Improvements:
  1. Add email notifications when an applicant is shortlisted or rejected.
  2. Allow HR to re-screen existing applications after editing job keywords.
  3. Store uploaded resumes in cloud storage (AWS S3 / Cloudinary) for persistence.
  4. Add resume parsing to extract structured data (name, education, skills).
  5. Allow HR to adjust NLP scoring weights through the settings page.
  6. Add a candidate comparison view to display two applicants side by side.
  7. Re-enable BERT semantic scoring on a higher-RAM production server.
  8. Add a feedback loop where HR shortlist/reject decisions train a future ML model.
